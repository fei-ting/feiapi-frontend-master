# 前端覆盖率工具与质量基线设计

## 1. 背景与目标

前端工程当前已有 Vitest、Vue Test Utils 和 32 个测试文件，但没有覆盖率提供器、覆盖率脚本或阈值配置。现有测试全部通过只能说明已执行场景没有回归，不能证明核心模块达到项目要求的 90% 行覆盖率。

本项整改在不改变业务行为的前提下完成以下目标：

1. 使用 V8 覆盖率提供器采集 Vue 和 TypeScript 运行时代码覆盖率。
2. 建立全量运行时代码基线，防止整体覆盖率下降。
3. 对本轮已经拆分并补测的四组核心模块设置更严格的独立门禁。
4. 将测试和覆盖率门禁接入 GitHub Actions，并覆盖长期维护的 `dev` 分支。
5. 保留可下载的 HTML 报告，便于定位未覆盖分支。

本项不引入 ESLint、不修改业务逻辑、不补充端到端测试，也不把类型源码正则检查计入核心模块覆盖率结论。

## 2. 方案选择

采用双层门禁：全量运行时代码使用首次有效采集结果作为质量基线，四组核心模块使用固定的高标准阈值。

不采用全量源码立即统一 90% 的原因是当前仍有后台管理和配额流程缺少完整测试，强制统一 90% 会把本项扩大为跨领域补测。也不采用单一低全局阈值，因为它会允许高覆盖文件抵消核心模块的不足。

## 3. 工具与命令

### 3.1 覆盖率提供器

新增与 Vitest `4.1.10` 完全一致的开发依赖：

```text
@vitest/coverage-v8@4.1.10
```

项目继续使用 Yarn 1 管理依赖，提交 `package.json` 和 `yarn.lock`，不生成或提交 `package-lock.json`。

### 3.2 脚本

保留现有快速测试命令：

```text
yarn test
```

新增质量门禁命令：

```text
yarn coverage
```

该命令运行全部 Vitest 测试、采集覆盖率并校验阈值。CI 不再额外重复执行一次 `yarn test`，因为 `yarn coverage` 已执行同一套测试。

## 4. 统计范围

覆盖率使用 `src/**/*.{ts,vue}` 作为运行时代码入口，并排除：

```text
src/**/*.d.ts
src/**/*.test.ts
src/**/__tests__/**
src/types/**
src/main.ts
```

排除理由：

1. 声明文件和 `src/types/**` 不包含可执行领域逻辑，类型正确性由 `vue-tsc` 负责。
2. 测试文件不能计入被测源码覆盖率。
3. `src/main.ts` 只负责应用启动和全局注册，不属于单元测试质量门禁范围。

普通页面、组件、组合式函数、Store、Service、工具函数和开发 Mock 均保留在全局统计中。未被任何测试导入的文件必须以零覆盖率进入结果，不能通过只统计已加载文件提高数字。

## 5. 双层阈值

### 5.1 全局基线

依赖安装后执行第一次有效覆盖率采集，分别读取语句、分支、函数和行覆盖率。每项全局阈值取该次结果的整数向下取整值，并写入 `vite.config.ts`。

例如首次行覆盖率为 `68.74%`，则全局行阈值固定为 `68%`。阈值不启用 `autoUpdate`，后续提升需要通过明确提交调整，防止工具自动改写配置。

首次采集必须满足以下条件才有效：

1. 32 个测试文件和 122 个现有测试全部执行。
2. 没有未处理的测试工作进程错误。
3. 覆盖率结果包含未被测试导入的运行时文件。
4. 报告同时生成终端摘要、HTML 和 JSON 摘要。

2026-07-17 首次有效采集执行了 32 个测试文件和 122 个测试，实测结果及固化阈值如下：

| 指标 | 实测结果 | 全局阈值 |
| --- | ---: | ---: |
| 语句 | 59.89% | 59% |
| 分支 | 52.54% | 52% |
| 函数 | 56.90% | 56% |
| 行 | 60.20% | 60% |

定向补充核心模块边界测试后，共执行 32 个测试文件和 133 个测试，最终整体覆盖率提升为语句 `62.17%`、分支 `56.72%`、函数 `59.45%`、行 `62.15%`，全局基线和全部核心模块门禁均通过。

### 5.2 核心模块门禁

以下四组模块代表第五部分已完成的组件化整改范围：

```text
src/components/dashboard/**
src/views/admin/DashboardView.vue

src/components/invoke/**
src/views/InterfaceInvokeView.vue

src/components/profile/**
src/views/profile/ProfileInfoView.vue

src/components/admin/doc/**
src/views/admin/InterfaceDocMaintenanceView.vue
```

每组路径分别执行聚合阈值校验：

| 指标 | 阈值 |
| --- | ---: |
| 行 | 90% |
| 语句 | 90% |
| 函数 | 85% |
| 分支 | 80% |

不启用全局 `perFile`，避免简单展示组件因单个模板编译分支导致不稳定失败；但每个路由父页面使用独立路径规则，因此不能由其他领域组件抵消其覆盖率。

若任一核心规则未达标，只补充对应模块的真实行为测试，包括加载、错误、空状态、输入事件、增删操作、保存校验和边界分支。禁止通过新增排除项、删除 `include` 路径或降低固定核心阈值绕过失败。

## 6. 报告配置

覆盖率报告器固定为：

```text
text
html
json-summary
```

报告目录继续使用 `coverage/`。该目录已被 `.gitignore` 排除，不提交生成物。

使用 `excludeAfterRemap`，确保 Vue 单文件组件经过源码映射后仍应用测试文件和声明文件排除规则。

## 7. CI 设计

更新 `.github/workflows/ci-frontend.yml`：

1. 工作流名称改为“前端 CI（类型检查 + 覆盖率 + 构建）”，移除当前没有实际执行的 Lint 字样。
2. `push` 和 `pull_request` 监听 `dev`、`main`、`master`。
3. 继续使用 Node.js 22、Yarn 缓存和 `yarn install --frozen-lockfile`。
4. 按“类型检查 -> 覆盖率门禁 -> 生产构建”的顺序执行。
5. 覆盖率或测试失败时命令返回非零状态并阻止后续构建。
6. 使用独立上传步骤保存 `coverage/`，保留 7 天。
7. 覆盖率失败时仍尝试上传已有报告；没有报告时忽略缺失文件，不能掩盖原失败状态。
8. 保留现有 `dist/` 构建产物上传步骤。

本项不新增 Lint 步骤。ESLint 依赖、旧 `.eslintrc.js` 和静态检查规则在后续独立整改中处理。

## 8. 失败与边界处理

1. 测试失败：覆盖率命令失败，不生成有效基线。
2. 覆盖率提供器或 `jsdom` 无法加载：视为环境失败，禁止使用 0% 或空报告配置阈值。
3. 全局阈值失败：说明整体覆盖率回退，必须补测或说明并显式调整设计和配置。
4. 核心阈值失败：只能补充对应核心模块测试，不能由其他文件覆盖率抵消。
5. HTML 报告上传失败：不改变测试和阈值检查结果，但 CI 应保留清晰日志。
6. Yarn 锁文件与依赖声明不一致：`--frozen-lockfile` 必须失败，避免 CI 使用漂移依赖。

## 9. 测试与验收

实现完成后执行：

```text
yarn install --frozen-lockfile
yarn test
yarn coverage
yarn typecheck
yarn build
```

验收标准：

1. 原 32 个测试文件、122 个测试继续全部通过；若新增测试，数量只增不减。
2. `yarn coverage` 在本地返回成功，并显示全局及核心模块阈值均通过。
3. `coverage/index.html` 和 `coverage/coverage-summary.json` 正常生成。
4. 未导入的运行时代码出现在覆盖率报告中。
5. `yarn.lock` 包含与 Vitest 版本一致的 V8 覆盖率依赖。
6. CI 配置监听 `dev`，并执行覆盖率门禁而不是重复普通测试。
7. `yarn typecheck` 和 `yarn build` 通过。
8. `git diff --check` 通过，`coverage/` 不进入 Git 暂存区。
